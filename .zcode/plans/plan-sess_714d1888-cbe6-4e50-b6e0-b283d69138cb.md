# 宠物表结构改造计划:从"小狗专属"到"多物种通用"

## 设计决策(已与用户确认)
- **物种建模**:单一 `breeds` 表 + `species` 字段;`pets` 表加 `species` 列。
- **物种特有字段**:用 JSON 列 `extra_attributes` 承载(猫的"纯室内"、兔的"笼养/散养"、鸟的"脚环号"等),保留通用字段的强类型。
- **新增通用字段**:来源类(`source`、`source_org`)。
- **迁移方式**:改 db crate 的 `#[derive(toasty::Model)]` 定义,用 `cargo run -p api --bin migrate -- migration generate --name multi_species` 生成新迁移文件(0001_xxx.sql)。

## 1. 新增 species 枚举(domain 层)

**文件**: `crates/domain/src/pet.rs`

新增 `Species` 枚举(参照现有 `Gender`/`NeuterStatus` 的 `as_str`/`from_str` 模式):

```rust
pub enum Species {
    Dog,    // 狗
    Cat,    // 猫
    Rabbit, // 兔
    Bird,   // 鸟
    Rodent, // 啮齿(仓鼠/豚鼠)
    Reptile,// 爬行
    Fish,   // 鱼
    Other,  // 其他
}
```
- `as_str`: "dog" / "cat" / "rabbit" / "bird" / "rodent" / "reptile" / "fish" / "other"
- `from_str`: 反向解析,未知值返回 `None`
- `from_breed_id` 辅助方法(可选):若犬种 id 前缀为 `brd_`,自动推断 species;不过为简单起见,创建时直接由调用方传 species。

新增 `AcquisitionSource` 枚举:
```rust
pub enum AcquisitionSource {
    Adoption,   // 领养
    Purchase,   // 购买
    Rescue,     // 救助
    Gift,       // 赠予
    Found,      // 拾得
    Bred,       // 自繁
    Other,      // 其他
}
```
- `as_str`: "adoption" / "purchase" / "rescue" / "gift" / "found" / "bred" / "other"

## 3. domain `Breed` 结构扩展

**文件**: `crates/domain/src/pet.rs`

`breeds` 表加 `species` 列以区分犬种/猫种/兔种等:

```rust
pub struct Breed {
    pub id: String,
    pub species: Species,       // 新增:该品种所属物种
    pub name: String,
    pub pinyin: String,
    pub initial: String,
    pub size_category: Option<String>,   // 改为 Option:猫/兔等无体型分类
    pub coat_type: Option<String>,       // 改为 Option:部分物种无毛型分类
    pub standard_weight_min: Option<f64>, // 改为 Option
    pub standard_weight_max: Option<f64>, // 改为 Option
    pub life_span_min: Option<i32>,      // 改为 Option
    pub life_span_max: Option<i32>,      // 改为 Option
    pub exercise_needs: Option<String>,  // 改为 Option
    pub icon: Option<String>,
    pub origin: Option<String>,
}
```

理由:犬种有"标准体重范围/寿命参考/运动需求/体型"等数据;猫、兔、鸟等物种通常无标准化数据,这些字段对它们没意义。改为 Option 让数据可为空,但保留同一张表统一查询。

## 5. db 层 Model 同步

**文件**: `crates/db/src/pet.rs`

### 5.1 `Pet` Model 加新列
```rust
#[derive(Debug, toasty::Model)]
#[table = "pets"]
pub struct Pet {
```

### 5.3 `From` mapper 更新
- `From<Pet> for domain_pet::Pet`:补 species/acquisition_source/source_org/adopted_at(extra_attributes 解析 `serde_json::from_str`)。
- `From<Breed> for domain_pet::Breed`:新增 species,其余 Option 字段直接透传。
- `PetRepository::create`:toasty::create! 补上新字段(extra_attributes 用 `serde_json::to_string` 序列化)。
- `PetRepository::update`:补 set_species / set_acquisition_source / set_source_org / set_adopted_at / set_extra_attributes。
- `BreedRepository::search`:加 species filter;size 仅在 species=dog 时应用。

## 6. 生成迁移

完成代码改动后,在项目根执行:

```bash
cargo run -p api --bin migrate -- migration generate --name multi_species
```

这会在 `toasty/migrations/` 生成 `0001_multi_species.sql`,内容大致为 `ALTER TABLE` 语句(pets 加列、breeds 改列 + 加 species)。然后:

```bash
cargo run -p api --bin migrate -- migration apply
```

把 schema 应用到 `data.db`。如果存量数据(犬种库 seed 数据)需要回填 species='dog',会在迁移生成后检查;若 toasty 不自动回填,我会手动在迁移 SQL 末尾补一句 `UPDATE breeds SET species='dog' WHERE species=''` 之类的兜底语句。

## 风险与缓解
1. **toasty 不支持 ALTER TABLE 改列类型**(NOT NULL → NULL):若自动生成失败,我会改为"新增列 + 迁移数据 + 删除旧列"的等价迁移,或必要时重写 `0000_initial.sql` 的 `breeds` 表(存量 seed 数据会重置)。若发生这种情况会先与你确认。
2. **JSON 列类型**:toasty 无原生 JSON,用 String 承载,在 domain 层做序列化。简单可靠,代价是数据库内无法直接查询 JSON 字段。本次可接受。
3. **`species` 必填但存量 pets 数据无值**:迁移末尾 `UPDATE pets SET species='dog' WHERE species IS NULL OR species=''` 兜底,因为原产品就是小狗。
